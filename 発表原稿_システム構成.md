# 発表原稿：システム構成 — サービスの組み合わせで実現する

> 想定時間：約4分
> ポイント：①単純な突き合わせを多様なサービスの組み合わせで実現した ②AI提案をそのまま採用しても、実装に関わる中で学びがあった
> ※ システム構成図（`システム構成図.svg`）と併せて使用

---

## ［導入：約40秒］

ここからは、作ったシステムの構成についてお話しします。

やりたいことは、実はとても単純です。「勤怠システムの打刻時間」と「工数システムの入力時間」を突き合わせて、ズレている人にメールで知らせる。たったこれだけです。

ところが、この単純な処理を実際に動かそうとすると、裏側ではかなり多くのサービスを組み合わせることになりました。今日はその構成図に沿って、何をどう組み合わせたかをご紹介します。

---

## ［構成の流れ：約1分40秒］

図の左上から、処理の順番に追っていきます。

まず**①**、社内のPCから、工数システムのデータベースに **SSH接続**してデータを取り出します。このデータベースは社内ネットワークの中にあるので、**②** 取り出したデータを一度 **Google Cloud のストレージ**に上げます。

**③** スケジューラが定期的に、**④** 中心となる **Cloud Functions** を呼び出します。ここが処理の心臓部で、片方では先ほどのストレージから工数データを読み、もう片方では **HRMOS の API** から勤怠データを取りに行きます。このとき、接続に必要なパスワードやキーは、コードに直接書かず **Secret Manager** という金庫から取り出します。

**⑤** 突き合わせてズレを見つけたら、**Microsoft の Graph API** を使ってメールを送ります。ここは認証に **Azure AD のアプリ登録**を使っています。**⑥** 実行した結果は **Firestore** というデータベースに記録します。

そして **⑦**、私自身は **Cloud Run** に乗せた管理画面から、手動実行したり結果を確認したりできます。

——このように、たった「2つの数字を比べる」ために、ストレージ、スケジューラ、関数、シークレット管理、データベース、外部API、メール基盤、管理画面と、**役割の違うサービスを一通り組み合わせる**ことになりました。

---

## ［なぜこんなに増えたか：約50秒］

なぜ単純な処理でサービスが増えたのか。理由は、現実の制約に一つずつ対応していった結果です。

クラウドから社内DBに直接つなげないから、社内PC経由でストレージに上げる。無料枠で収めたいから、構成を工夫する。パスワードを安全に持ちたいから、シークレット管理を使う。メールを確実に送りたいから、Microsoft の基盤を使う。

一つひとつは当たり前の判断ですが、積み重ねると「単純な処理＝単純な構成」にはならない。むしろ、**身近なクラウドサービスを組み合わせるだけで、追加費用ほぼゼロで実用的なシステムが作れる**、ということでもあります。

---

## ［AIに任せても、学びはあった：約50秒］

もう一つお伝えしたいのが、この構成のほとんどは **AI の提案をそのまま採用して作った**、という点です。「この構成でいきましょう」と出てきたものに、私は基本「OK」と答えて進めました。

でも、丸ごとお任せだったかというと、そうではありませんでした。実際に手を動かして実装に関わる中で、**自分が知らなかったことを知るきっかけ**が何度もあったんです。

たとえば、社内DBにはSSHトンネルを通すという発想。シークレットの保管にも無料枠の上限があること。メール送信には Azure 側でアプリを登録して権限を与える必要があること。Cloud Run の認証の考え方。——どれも、言われるまで知らなかったり、ぼんやりとしか分かっていなかったことです。

AI に提案してもらって作る場合でも、**実装に関われば、その過程自体が学びになる**。ここは実感としてお伝えしたいところです。

---

## ［まとめ：約30秒］

まとめます。今回のシステムは、やること自体は単純な突き合わせですが、**多様なクラウドサービスを組み合わせる**ことで、無料枠の中で実現できました。

そして、AI に構成を提案してもらってそのまま作ったとしても、実装に関わる中で新しい知識に出会える。サービスを組み合わせて何かを作る経験は、それ自体に価値がある、と感じています。

以上です。

## 付録：登場するサービス一覧

| 区分 | サービス | 役割 |
|---|---|---|
| 社内 | 社内PC（取得スクリプト） | SUMTIME DBからSSHでデータ取得 |
| 社内 | SUMTIME DB（PostgreSQL） | 工数データの取得元 |
| GCP | Cloud Storage | 工数データ（master）の置き場 |
| GCP | Cloud Scheduler | 定期実行のトリガー |
| GCP | Cloud Functions | 突き合わせ処理の中心 |
| GCP | Secret Manager | 認証情報の保管 |
| GCP | Firestore | 設定・実行結果の保存 |
| GCP | Cloud Run | 管理UIのホスティング |
| 外部 | HRMOS API | 勤怠データの取得元 |
| 外部 | Azure AD / Microsoft Graph | メール送信（認証＋送信API） |
| 外部 | 共有メールボックス | 送信元アドレス |
