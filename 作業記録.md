## 1. 要件ヒアリング結果

以下の要件が会話を通じて確定した。

### 1.1 基本要件

| 項目 | 決定内容 |
|---|---|
| 目的 | HRMOS勤怠の勤怠時間と自社工数管理システムの稼働実績の差異検出 |
| 実行頻度 | 週1回（毎週月曜 9:00 JST） |
| インフラ | Google Cloud Platform（GCP） |
| 対象人数 | 約30名 |
| 通知先 | 差異がある本人にも通知（+ 管理者はチャンネルで一覧確認） |

| 項目 | 決定内容 |
|---|---|
| 突き合わせキー | 社員番号（両システム共通） |
| 差異の閾値 | 30分以上 |
| 集計単位 | 社員ごと・週単位 |

### 1.3 通知仕様

| 項目 | 決定内容 |
|---|---|
| 通知手段 | Microsoft Teams（Power Automate Workflows Webhook） |
| 通知方式 | 差異のある社員ごとに1件ずつAdaptive Cardを投稿 |
| メンション | 各カード内で本人を `<at>` タグでメンション |

## 2. 技術調査・確認事項

### 2.1 GCP 無料枠の確認

GCP の各サービスが無料枠内で運用可能であることを確認した。

| サービス | 無料枠 | 想定使用量 | 判定 |
|---|---|---|---|
| Cloud Scheduler | 3ジョブ/アカウント（登録数） | 1ジョブ | ✅ 範囲内 |
| Cloud Functions 呼び出し | 200万回/月 | 約4回/月 | ✅ 範囲内 |
| Cloud Functions 実行時間 | 400,000 GB秒/月 | 約0.5 GB秒/月 | ✅ 範囲内 |
| BigQuery クエリ | 1 TB/月 | 数MB/月 | ✅ 範囲内 |
| BigQuery ストレージ | 10 GB | 数KB〜数MB | ✅ 範囲内 |

**注意事項：**
- Cloud Schedulerの「3ジョブ」は実行回数ではなく登録ジョブ数。週1実行でも登録は1ジョブなので無料枠内。
- Cloud Functionsデプロイ時にCloud Build / Artifact Registryが使われるが、頻繁なデプロイでなければ実質無料。
- 念のためGCPの請求アラートを$1程度に設定することを推奨。

### 2.2 Teams Incoming Webhook の調査

- 従来の Office 365 コネクタ（Incoming Webhook）はMicrosoftが廃止を進めている。
- 廃止期限は2026年4月30日まで延長されている。
- **新規構築では Power Automate Workflows の Webhook トリガーを使用すべき。**
- Workflows でも URL に POST するだけで通知可能。URLの形式が異なるのみ。

### 2.3 HRMOS勤怠 API の調査（2026-03-25）

- 公式ドキュメント: https://ieyasu.co/docs/api.html
- OpenAPI定義: https://ieyasu.co/docs/openapi.json
- 認証方式: APIキー → Basic認証でトークン取得（有効期限24時間、延長API あり）
- 使用エンドポイント: `GET /work_outputs/monthly/{month}`
- 必要なフィールドはすべて取得可能であることを確認（社員番号=`number`、勤務日=`day`、出勤=`start_at`、退勤=`end_at`、実労働時間=`actual_working_hours`）
- 休憩時間は `break_1_start_at`/`break_1_end_at` 等の差分から算出が必要
- レート制限: 過去60秒累計で制限（具体的数値は非公開、超過時 HTTP 429）
- APIキー取得には有料プランが必要（管理者権限でシステム設定から発行）

### 2.4 工数管理システムのデータ取得方式（2026-03-25）

- API経由ではなくDB直接アクセスで取得する方針に決定
- テーブル構造・接続情報は別途確認予定

### 2.5 Teams メンション機能の調査

- Adaptive Card 内で `<at>` タグと `msteams.entities` を使うことで特定ユーザーへのメンションが可能。
- メンションには Azure AD の UPN（メールアドレス）または Object ID が必要。
- 1メッセージあたりのメンション上限は20ユーザー。1人1カード方式なら制約に抵触しない。

## 3. 設計

### 3.1 システム構成図

以下の構成で合意した。

```
Cloud Scheduler → Cloud Functions → HRMOS勤怠 API
                                  → 工数管理システム DB
                                  → BigQuery（ログ蓄積・任意）
                                  → Teams Webhook（通知）
```

### 3.2 通知方式の決定

2パターンを検討し、パターン1を採用。

- ✅ **パターン1（採用）：** 差異がある社員ごとに個別のAdaptive Cardを投稿し、各カードで本人をメンション
- ❌ パターン2：全員の差異を1枚のカードにまとめて複数人を同時メンション

採用理由：本人にとって自分の差異だけ見えてわかりやすい。

### 3.3 アカウント構成の決定

小規模な業務改善ツールとして個人主導で開始する方針を決定。

| 環境 | アカウント種別 |
|---|---|
| GCP | 個人 Google アカウント |
| Microsoft Teams | 社内アカウント |

**認識しているリスク：**

- GCPアカウント所有者の退職・閉鎖時にシステム停止
- 請求責任が個人に帰属
- 社内APIキーを個人GCPに保存する点のセキュリティ確認が必要
- Teams Webhook は作成ユーザーに紐づくため、共同所有者の設定が必要

**方針：** 運用が定着した段階で会社管理のGCPアカウントへ移行する。
