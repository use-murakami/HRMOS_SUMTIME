# 仕様駆動開発の流れ — 勤怠・工数突き合わせシステム

> 本ドキュメントは、本プロジェクトで実際に辿った仕様駆動開発のプロセスを記録したもの。
> （※後でブラッシュアップ予定）

---

## 全体像（1行サマリー）

**要件ヒアリング → 実機検証で前提を確定 → 仕様書に固定（v0.1→v0.6）→ 実データ調査 → テスト付き実装 → 段階デプロイ → 安全ガード下で実データ検証 → 本番移行（予定）**

仕様書のバージョンが、確定した事実とともに育っていったのが最大の特徴。

---

## フェーズ別の流れ

### ① 要件ヒアリング・初期仕様（〜2026-03）

- **目的**：HRMOS勤怠と工数管理システム(SUMTIME)の差異を検出し、本人にメール通知
- **初期の前提（後に多くが覆る）**：社員番号で突合／週単位／30分閾値／Teams通知
- 成果物：仕様書 **v0.1**

### ② 実機検証で「外部の事実」を確定（2026-03-31〜04-20）★最重要

検証スクリプト（`verify_*.py`）で外部要素を実際に叩き、**想像していた前提を事実で上書き**した。

| 検証対象 | 判明した事実 | 仕様への影響 | 層 |
|---|---|---|---|
| HRMOS API (`verify_api.py`) | 認証方式・実働時間フィールド形式・email存在 | 仕様確定 | — |
| SUMTIME DB (`verify_db.py`) | **社員番号が存在しない** | 突合キーを**メールアドレス**に変更 | ロジック |
| ネットワーク到達性 | **クラウドから社内DBに到達不可（プライベートIP）** | **構成を「社内PC→Cloud Storage→Cloud Functions読込」に変更** | アーキテクチャ |
| Teams送信 (`verify_teams_dm.py`) | サービスプリンシパルは1:1チャットに参加不可（HTTP 403） | 通知を**メール（Graph API Mail.Send）**に変更 | 通知手段 |
| コスト | Power Automate Premium（~$15/月）／IAP（~$18/月）が必要 | 無料の代替（Graph API直叩き／Flask Basic認証）へ | コスト |

→ 成果物：仕様書 **v0.2〜v0.3**（通知方式変更・付録に断念経緯を記録）

### ③ 実データ調査で仕様を精緻化（2026-04-20）

- 調査スクリプト群（`hrmos_monthly_hours.py` / `sumtime_monthly_hours.py` / `check_discrepancy*.py`）で実際の乖離状況を確認
- **現実のデータから仕様が育った例**：
  - 「月例」を含む勤務区分は突合対象外
  - HRMOS実働時間は日別に15分切り捨て（SUMTIMEが15分単位入力のため）

### ④ インフラ構築・認証情報整理（2026-04-24〜05-21）

- GCPプロジェクト作成（`kintai-kosu-notification`）・請求アラート（1円）・必要API有効化
- Secret Manager：当初**11個** → **3個のJSONシークレットに統合**（無料枠対応・呼び出し側は後方互換維持）

### ⑤ テスト付き実装（2026-05-08〜）

- モジュール単位で「実装＋テスト」をセットで作成
- 主なモジュール：`config` / `secret_manager` / `hrmos_client` / `storage_client` / `reconciler` / `email_client` / `email_builder` / `main`
- テスト累計：**76件 → 最終193件**（全パス）

### ⑥ 管理UIを初期リリースに格上げ・仕様確定（2026-05-28〜29）

- 仕様書 **v0.5 → v0.6 最終確定**（管理UI 5画面・Firestoreスキーマ・Basic認証）
- 実装方針を明文化：**ドラフト仕様 → 技術検証 → 最終確定 → 実装** の順

### ⑦ 段階デプロイ（2026-06-05）

- Cloud Functions / 管理UI(Cloud Run) / Cloud Scheduler を順にデプロイ
- Cloud Scheduler は**全ジョブを停止(PAUSED)状態で登録**（誤発火防止）
- **3層安全ガード**：通知対象OFF＋UIトグルロック(TESTING_MODE)＋メール強制転送(OVERRIDE_EMAIL_TO)

### ⑧ 安全ガード下での実データ検証（Phase 7・進行中）

- プレビュー → 実メール送信テスト（管理者宛のみ）成功
- 検証中に判明したニーズを随時反映：
  - 過去月参照のための**マスターファイル蓄積方式**
  - 即時実行の**「管理者のみ」対象範囲スイッチ**
  - プレビューの**差異合計（絶対値）行**

### ⑨ 本番移行（Phase 8・来月予定）

- Cloud Functions 環境変数を本番用に変更（`OVERRIDE_EMAIL_TO=""` / `TESTING_MODE=false`）
- Cloud Scheduler を有効化(resume)

## この流れを貫く「仕様駆動の型」

1. **想像で決めない** — 外部の前提は実機で確定させてから仕様に書く
2. **仕様書は記録媒体** — 決定が変わるたびに更新（v0.1→v0.6）。版数が増えるのは健全
3. **不確実性が高い順に潰す** — 動くか・届くか・お金がかかるか を先に検証
4. **テストを安全網に** — 後の大改修も怖くない（最終193件）
5. **可逆・段階的に** — 停止状態で登録し、安全ガード下で実データ検証
6. **AIの提案も問い直す** — 「本当に問題ない？」で漏れ・方向違いを実装前に吸収
