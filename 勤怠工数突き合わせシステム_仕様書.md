## 1. 概要

### 1.1 目的

HRMOS勤怠で管理される勤怠データ（出退勤時間）と、自社開発の工数管理システムで記録される稼働実績データを週次で突き合わせ、差異が一定の閾値を超える社員を検出し、Microsoft Teams 経由で本人および管理者に通知する。

### 1.2 背景

勤怠時間と工数実績の乖離は、打刻漏れ・工数入力漏れ・実態との不一致などを示唆する。手動での照合は工数がかかるため、自動化して早期に差異を把握できる仕組みを構築する。

### 1.3 対象ユーザー

- 対象社員：約30名
- 通知先：差異が検出された本人 + 管理者

## 2. システム構成

### 2.1 インフラストラクチャ

Google Cloud Platform（GCP）上に構築する。すべて無料枠内での運用を想定する。

### 2.2 アカウント構成

小規模な業務改善ツールとして個人主導で開始する。

| 環境 | アカウント種別 | 備考 |
|---|---|---|
| GCP | 個人 Google アカウント | 無料枠内で運用。請求責任は個人に帰属 |
| Microsoft Teams | 社内アカウント | Power Automate Workflows Webhook を使用 |

**運用上のリスクと対策：**

- GCP アカウント所有者の退職・アカウント閉鎖時にシステムが停止するリスクがある。運用が定着した段階で会社管理のGCPアカウントへの移行を検討する。
- Secret Manager に社内システムの API キーを保存するため、社内 IT 管理者への事前確認を推奨する。
- Teams 側の Webhook は作成ユーザーに紐づくため、共同所有者（co-owner）を設定しておくことを推奨する。

**将来の移行方針：**

本システムの有効性が確認された場合、以下の移行を行う。

1. 会社管理の GCP 組織アカウントへプロジェクトを移行
2. Secret Manager の認証情報を再設定
3. Webhook の所有者をサービスアカウントまたは共有アカウントに変更

### 2.3 使用サービス

| サービス | 役割 | 無料枠 |

## 3. データ仕様

### 3.1 突き合わせキー

- **社員番号**（両システム共通）

### 3.2 HRMOS勤怠 API 連携仕様

**基本情報:**

| 項目 | 値 |
|---|---|
| ベースURL | `https://ieyasu.co/api/use/v1/` |
| ドキュメント | https://ieyasu.co/docs/api.html |
| OpenAPI定義 | https://ieyasu.co/docs/openapi.json |

#### 3.2.1 認証フロー

2段階のトークン認証方式。

**Step 1: トークン取得**

```
GET /authentication/token
Authorization: Basic base64({secret_key}:)
```

- Secret Key をユーザー名、パスワードは空文字で Basic 認証
- レスポンス: `{ "token": "...", "expired_at": "..." }`
- トークン有効期限: **24時間**

**Step 2: API呼び出し**

```
GET /work_outputs/monthly/2026-03
Authorization: Token {token}
```

**トークン延長（必要に応じて）:**

```
PUT /authentication/postpone
Authorization: Token {token}
```

- リクエスト時点から24時間に延長

> ※ 週1回の実行のため、毎回トークンを新規取得する方式とする（延長は使用しない）。

#### 3.2.2 使用エンドポイント

**① 勤怠データ取得: `GET /work_outputs/monthly/{month}`**

| パラメータ | 位置 | 必須 | 型 | 説明 |
|---|---|---|---|---|
| `month` | パス | ✅ | string | 対象月（YYYY-MM形式） |
| `user_id` | クエリ | - | integer | 特定ユーザーのみ取得 |
| `limit` | クエリ | - | integer | 1ページあたりの件数（デフォルト25、最大100） |
| `page` | クエリ | - | integer | ページ番号（デフォルト1） |

レスポンスヘッダ:
- `X-Total-Count`: 総データ件数
- `X-Total-Page`: 総ページ数

**② ユーザー一覧取得: `GET /users`**

| パラメータ | 位置 | 必須 | 型 | 説明 |
|---|---|---|---|---|
| `limit` | クエリ | - | integer | 1ページあたりの件数（デフォルト25、最大100） |
| `page` | クエリ | - | integer | ページ番号（デフォルト1） |

#### 3.2.3 レスポンスフィールドと内部マッピング

**勤怠データ（WorkOutput）— 使用するフィールド:**

| HRMOS フィールド | 型 | 値の例 | 内部フィールド | 変換処理 |
|---|---|---|---|---|
| `number` | string | `"A0001"` | employee_id | そのまま使用 |
| `day` | string | `"2026-03-16"` | date | date型に変換 |
| `start_at` | string | `"08:00"` | clock_in | 参考情報として保持 |
| `end_at` | string | `"17:00"` | clock_out | 参考情報として保持 |
| `total_break_time` | string | `"1:00"` | break_minutes | `"H:MM"` → 分に変換 |
| `actual_working_hours` | string | `"8:00"` | working_minutes | `"H:MM"` → 分に変換 |

> ※ 時間フィールドはすべて `"H:MM"` 形式の文字列。分への変換が必要。
> 例: `"8:00"` → 480分、`"1:30"` → 90分

**勤怠データ（WorkOutput）— 参照用フィールド:**

| HRMOS フィールド | 説明 | 用途 |
|---|---|---|
| `user_id` | HRMOS内部ユーザーID | APIのuser_idフィルタに使用 |
| `full_name` | 氏名 | ログ・デバッグ用 |
| `wday` | 曜日 | 対象期間のフィルタ確認用 |
| `segment_display_title` | 勤務区分（"稼働"、"有休"等） | 有休等の除外判定に使用可能 |
| `next_day_start` | 翌日フラグ（出勤） | 夜勤判定（0=当日、1=翌日） |
| `next_day_end` | 翌日フラグ（退勤） | 夜勤判定（0=当日、1=翌日） |

**ユーザーデータ（User）— 使用するフィールド:**

| HRMOS フィールド | 型 | 値の例 | 内部フィールド | 用途 |
|---|---|---|---|---|
| `number` | string | `"A0001"` | employee_id | 突き合わせキー |
| `last_name` | string | `"徳川"` | display_name（姓） | Teams通知の表示名 |
| `first_name` | string | `"家康"` | display_name（名） | Teams通知の表示名 |
| `email` | string | `"ieyasu@example.com"` | email | Teamsメンション用UPN |

#### 3.2.4 データ取得の処理フロー

```
1. トークン取得
   POST Basic認証 → token を取得

2. 対象月の特定
   - 前週月曜〜日曜が同一月内 → 1ヶ月分を取得
   - 前週が月をまたぐ場合 → 2ヶ月分を取得

3. 勤怠データ取得（ページネーション）
   GET /work_outputs/monthly/{month}?limit=100&page=1
   → X-Total-Page を確認し、全ページを順次取得

4. 対象期間でフィルタ
   取得データから前週月曜〜日曜の `day` のレコードのみ抽出

5. ユーザー一覧取得（ページネーション）
   GET /users?limit=100&page=1
   → 社員番号・氏名・メールアドレスを取得

6. 時間変換
   `actual_working_hours`（"H:MM"形式）→ 分（integer）に変換
```

#### 3.2.5 ページネーション見積もり

| 項目 | 見積もり |
|---|---|
| 対象社員数 | 約30名 |
| 1ヶ月の勤怠レコード数 | 約30名 × 22営業日 = 約660件 |
| limit=100 でのページ数 | 7ページ |
| 月またぎの場合 | 最大14ページ（2ヶ月分） |
| ユーザー一覧 | 1ページ（30名 < limit=100） |
| 1回の実行あたり合計リクエスト数 | 約10〜17リクエスト（認証1 + 勤怠7〜14 + ユーザー1） |

#### 3.2.6 レート制限への対応

- 過去60秒のリクエスト累計で制限（具体的数値は非公開）
- 超過時: HTTP 429（レスポンスボディなし）
- 対策: ページネーションのリクエスト間に **1秒のウェイト** を挿入
- 429 受信時: **5秒待機してリトライ**（最大3回）

### 3.3 工数管理システムから取得するデータ

**接続方式:** データベース直接アクセス（詳細は別途確認）

| フィールド | 型 | 説明 |
|---|---|---|
| employee_id | string | 社員番号 |
| date | date | 作業日 |
| total_minutes | integer | 日別の工数合計（分） |
| project_code | string | プロジェクトコード（参考情報） |

> ※ 実際のカラム名・テーブル構造は工数管理システムのDB定義に準ずる。導入時にマッピング定義を行う。

### 3.4 社員マスタ（メンション用）

Teams でのメンション通知のために、社員番号とTeamsアカウント（UPN / メールアドレス）の紐付けが必要。

| フィールド | 型 | 説明 |
|---|---|---|
| employee_id | string | 社員番号 |
| display_name | string | 表示名 |
| email | string | メールアドレス（= Teams UPN） |

**管理方法の選択肢：**

- A) 工数管理システムのDBから取得
- **B) HRMOS勤怠の `/users` APIから取得 ← 推奨**
- C) Cloud Functions 内に環境変数 or JSON ファイルとして保持
- D) BigQuery テーブルとして管理

> HRMOS `/users` APIで `number`（社員番号）、`last_name`/`first_name`（氏名）、`email`（メールアドレス=Teams UPN）がすべて取得可能。
> 勤怠データ取得と同じトークンで呼び出せるため、追加の認証設定が不要。
> ※ 具体的な管理方法は実装フェーズで確定する。

## 4. 突き合わせロジック

### 4.1 対象期間

- 前週の月曜日〜日曜日（7日間）
- 実行タイミング：毎週月曜 9:00 JST

### 4.2 集計単位

- 社員ごとに**週単位**で集計する
- 勤怠側：対象期間の `working_minutes` を合算
- 工数側：対象期間の `total_minutes` を合算

### 4.3 差異の算出

```
差異（分） = 勤怠合計時間（分） − 工数合計時間（分）
```

### 4.4 通知閾値

- **|差異| ≧ 30分** の場合に通知対象とする
- 差異の方向（勤怠 > 工数、勤怠 < 工数）も通知に含める

### 4.5 差異パターン

| パターン | 意味 | 表示例 |
|---|---|---|
| 勤怠 > 工数 | 工数の入力漏れの可能性 | 勤怠: 40h / 工数: 38h（差異: +2.0h） |
| 勤怠 < 工数 | 打刻漏れ or 工数過大入力の可能性 | 勤怠: 35h / 工数: 40h（差異: -5.0h） |
| 勤怠データなし | 勤怠未打刻 | 勤怠データがありません |
| 工数データなし | 工数未入力 | 工数データがありません |

## 5. 通知仕様

### 5.1 通知チャネル

- Microsoft Teams の指定チャンネルに投稿
- Power Automate Workflows の Webhook トリガーを使用

### 5.2 通知方式

- 差異が検出された社員ごとに **1件ずつ Adaptive Card を投稿**
- 各カード内で **本人をメンション**（`<at>` タグ + `msteams.entities`）
- 管理者向けにはチャンネル上で一覧として確認可能

### 5.3 Adaptive Card 構成

```json
{
  "type": "message",
  "attachments": [{
    "contentType": "application/vnd.microsoft.card.adaptive",
    "content": {
      "type": "AdaptiveCard",
      "version": "1.4",
      "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
      "body": [
        {
          "type": "TextBlock",
          "size": "Medium",
          "weight": "Bolder",
          "text": "⚠ 勤怠・工数 差異検出"
        },
        {
          "type": "TextBlock",
          "text": "<at>山田太郎</at>",
          "wrap": true
        },
        {
          "type": "FactSet",
          "facts": [
            { "title": "対象期間", "value": "2026/03/16 〜 2026/03/22" },
            { "title": "勤怠時間", "value": "40.0h" },
            { "title": "工数時間", "value": "37.5h" },
            { "title": "差異", "value": "+2.5h（工数入力漏れの可能性）" }
          ]
        }
      ],
      "msteams": {
        "entities": [{
          "type": "mention",
          "text": "<at>山田太郎</at>",
          "mentioned": {
            "id": "yamada.taro@example.com",
            "name": "山田太郎"
          }
        }]
      }
    }
  }]
}
```

### 5.4 サマリー通知（任意）

全社員の突き合わせ完了後に、管理者向けのサマリーカードを1件投稿する。

- 対象期間
- 対象社員数
- 差異検出人数
- 差異なし人数
- データ欠損人数（勤怠 or 工数が取得できなかった社員）

### 5.5 通知の制約

- 1メッセージあたりのメンション上限：20ユーザー
  - 1人1カード方式のため本制約には抵触しない

## 7. エラーハンドリング

### 7.1 API接続エラー

| エラー種別 | 対応 |
|---|---|
| HRMOS API 接続失敗 | 最大3回リトライ（指数バックオフ）。失敗時は管理者にTeams通知 |
| 工数管理DB 接続失敗 | 最大3回リトライ（指数バックオフ）。失敗時は管理者にTeams通知 |
| Teams Webhook 送信失敗 | 最大3回リトライ。失敗時はCloud Loggingにエラーログ記録 |
| レート制限（HTTP 429） | Retry-After ヘッダーに従いリトライ |

### 7.2 データ異常

| 異常種別 | 対応 |
|---|---|
| 社員番号が片方のシステムにしか存在しない | 通知カードに「データ欠損」として表示 |
| 勤怠時間が24hを超過 | 異常値として警告付きで通知 |
| 工数時間が負の値 | 異常値として警告付きで通知 |

### 7.3 ログ出力

- すべての実行結果を Cloud Logging に出力
- ログレベル：INFO（正常）、WARNING（データ異常）、ERROR（API障害）

## 8. セキュリティ

### 8.1 認証情報の管理

- HRMOS API キー → Secret Manager
- 工数管理システム DB 接続情報 → Secret Manager
- Teams Webhook URL → Secret Manager

### 8.2 通信

- すべての外部API通信は HTTPS
- Cloud Functions には最小権限の IAM ロールを付与

### 8.3 データ保護

- 個人情報（社員番号、氏名、メールアドレス）を含むため、BigQuery のアクセス権限は管理者に限定
- ログに認証情報を出力しない

## 10. テスト方針

### 10.1 単体テスト

| テスト対象 | テスト内容 |
|---|---|
| HRMOS API クライアント | モックAPIに対するリクエスト・レスポンスのパース |

| テスト対象 | テスト内容 |
