## 1. 概要

### 1.1 目的

HRMOS勤怠で管理される勤怠データ（出退勤時間）と、自社開発の工数管理システムで記録される稼働実績データを日次または週次で突き合わせ、差異が一定の閾値を超える社員を検出し、**メール**で本人に通知する。管理者には毎回実行後にサマリーメールを送信し、エラー発生時は別途エラーメールを送信する。

### 1.2 背景

勤怠時間と工数実績の乖離は、打刻漏れ・工数入力漏れ・実態との不一致などを示唆する。手動での照合は工数がかかるため、自動化して早期に差異を把握できる仕組みを構築する。

### 1.3 対象ユーザー

- 対象社員：HRMOS 19名 / SUMTIME 14名（2026-03時点）
- SUMTIME未登録5名：HRMOSに打刻がある場合は工数0として差異検出・通知。打刻がない日はスキップ
- 通知先：差異が検出された本人 + 管理者（毎回サマリー。エラー時は別途エラーメール）

## 2. システム構成

### 2.1 インフラストラクチャ

Google Cloud Platform（GCP）上に構築する。すべて無料枠内での運用を想定する。

### 2.2 アカウント構成

小規模な業務改善ツールとして個人主導で開始する。

| 環境 | アカウント種別 | 備考 |
|---|---|---|
| GCP | 個人 Google アカウント | 無料枠内で運用。請求責任は個人に帰属 |
| Microsoft 365 | 社内アカウント | Microsoft Graph API（Azure AD アプリ権限）でメール送信 |

**運用上のリスクと対策：**

- GCP アカウント所有者の退職・アカウント閉鎖時にシステムが停止するリスクがある。運用が定着した段階で会社管理のGCPアカウントへの移行を検討する。
- Secret Manager に社内システムの API キーを保存するため、社内 IT 管理者への事前確認を推奨する。
- Azure AD アプリのクライアントシークレットには有効期限がある（最大2年）。期限切れ前に更新が必要。
- 将来的にTeams通知への移行を検討する場合は、Azureサブスクリプションの取得が前提条件となる（付録B参照）。

**将来の移行方針：**

本システムの有効性が確認された場合、以下の移行を行う。

1. 会社管理の GCP 組織アカウントへプロジェクトを移行
2. Secret Manager の認証情報を再設定
3. Azure AD アプリの所有者を複数名に設定（退職リスク対策）

## 3. データ仕様

### 3.0 タイムゾーンの前提

すべての時刻データは **JST（Asia/Tokyo）** として扱う。

| データソース | 時刻形式 | タイムゾーン |
|---|---|---|
| HRMOS API `actual_working_hours` | "H:MM" 文字列 | JST（日本向けサービス） |
| HRMOS API `day` | "YYYY-MM-DD" 文字列 | JST |
| SUMTIME DB `start_at` | timestamp without time zone | JST（サンプルデータから確認） |

> ※ 両システムともJSTのため、日付単位の突き合わせで日付境界の不一致は発生しない。
> ※ Cloud Functions の実行環境では `TZ=Asia/Tokyo` を環境変数で設定する。

### 3.1 突き合わせキー

- **メールアドレス（email）**（両システム共通: `@use-eng.co.jp` ドメイン）

> ※ SUMTIMEに社員番号フィールドが存在しないことを実機確認済み（2026-03-31）。メールアドレスを共通キーとして使用する。

### 3.2 HRMOS勤怠 API 連携仕様

**基本情報:**

| 項目 | 値 |
|---|---|
| ベースURL | `https://ieyasu.co/api/{company_url}/v1/` |
| company_url | `use`（ログインURL `https://p.ieyasu.co/use/` から特定） |
| ドキュメント | https://ieyasu.co/docs/api.html |
| OpenAPI定義 | https://ieyasu.co/docs/openapi.json |

#### 3.2.1 認証フロー

2段階のトークン認証方式。

**Step 1: トークン取得**

```
GET /authentication/token
Authorization: Basic {secret_key}
Content-Type: application/json
```

- Secret Key は発行時点で Base64 エンコード済み。二重エンコードせずそのまま使用する
- `Authorization: Basic {secret_key}` の形式でリクエスト
- レスポンス: `{ "token": "...", "expired_at": "..." }`
- トークン有効期限: **24時間**

**Step 2: API呼び出し**

```
GET /work_outputs/monthly/2026-03
Authorization: Token {token}
```

**トークン延長（必要に応じて）:**

```
PUT /authentication/postpone
Authorization: Token {token}
```

- リクエスト時点から24時間に延長

> ※ 毎日の実行のため、毎回トークンを新規取得する方式とする（延長は使用しない）。

#### 3.2.2 使用エンドポイント

**① 勤怠データ取得: `GET /work_outputs/monthly/{month}`**

| パラメータ | 位置 | 必須 | 型 | 説明 |
|---|---|---|---|---|
| `month` | パス | ✅ | string | 対象月（YYYY-MM形式） |
| `user_id` | クエリ | - | integer | 特定ユーザーのみ取得 |
| `limit` | クエリ | - | integer | 1ページあたりの件数（デフォルト25、最大100） |
| `page` | クエリ | - | integer | ページ番号（デフォルト1） |

レスポンスヘッダ:
- `X-Total-Count`: 総データ件数
- `X-Total-Page`: 総ページ数

**② ユーザー一覧取得: `GET /users`**

| パラメータ | 位置 | 必須 | 型 | 説明 |
|---|---|---|---|---|
| `limit` | クエリ | - | integer | 1ページあたりの件数（デフォルト25、最大100） |
| `page` | クエリ | - | integer | ページ番号（デフォルト1） |

#### 3.2.3 レスポンスフィールドと内部マッピング

**勤怠データ（WorkOutput）— 使用するフィールド:**

| HRMOS フィールド | 型 | 値の例 | 内部フィールド | 変換処理 |
|---|---|---|---|---|
| `number` | string | `"A0001"` | employee_id | そのまま使用 |
| `day` | string | `"2026-03-16"` | date | date型に変換 |
| `start_at` | string | `"08:00"` | clock_in | 参考情報として保持 |
| `end_at` | string | `"17:00"` | clock_out | 参考情報として保持 |
| `total_break_time` | string | `"1:00"` | break_minutes | `"H:MM"` → 分に変換 |
| `actual_working_hours` | string | `"8:00"` | working_minutes | `"H:MM"` → 分に変換 |

> ※ 時間フィールドはすべて `"H:MM"` 形式の文字列。分への変換が必要。
> 例: `"8:00"` → 480分、`"1:30"` → 90分

**勤怠データ（WorkOutput）— 参照用フィールド:**

| HRMOS フィールド | 説明 | 用途 |
|---|---|---|
| `user_id` | HRMOS内部ユーザーID | APIのuser_idフィルタに使用 |

| `next_day_start` | 翌日フラグ（出勤） | 夜勤判定（0=当日、1=翌日） |
| `next_day_end` | 翌日フラグ（退勤） | 夜勤判定（0=当日、1=翌日） |

**ユーザーデータ（User）— 使用するフィールド:**

| HRMOS フィールド | 型 | 値の例 | 内部フィールド | 用途 |
|---|---|---|---|---|
| `email` | string | `"ieyasu@use-eng.co.jp"` | email | **突き合わせキー** + メール送信先 |
| `last_name` | string | `"徳川"` | display_name（姓） | メール本文の宛名表示 |
| `first_name` | string | `"家康"` | display_name（名） | メール本文の宛名表示 |
| `number` | string | `"A0001"` | employee_id | 参考情報（ログ用） |

#### 3.2.4 データ取得の処理フロー

```
1. トークン取得
   GET Basic認証 → token を取得

2. 対象期間の特定
   - 実行日前日から `period_days` 日前まで（デフォルト: 14日）
   - 期間が月をまたぐ場合 → 複数月分を取得

3. 勤怠データ取得（ページネーション）
   GET /work_outputs/monthly/{month}?limit=100&page=1
   → X-Total-Page を確認し、全ページを順次取得

4. 対象期間・勤務区分でフィルタ
   取得データから対象期間（過去2週間）の `day` のレコードのうち、
   `segment_display_title` が突き合わせ対象の勤務区分に該当するもののみ抽出

5. ユーザー一覧取得（ページネーション）
   GET /users?limit=100&page=1
   → email・氏名を取得（突き合わせキーとして使用）

6. 時間変換
   `actual_working_hours`（"H:MM"形式）→ 分（integer）に変換
   例: "8:00" → 480分、"1:30" → 90分
```

#### 3.2.5 突き合わせ対象の勤務区分

`segment_display_title` の値が以下のいずれかに該当するレコードのみ、SUMTIMEとの突き合わせ対象とする。

| # | 勤務区分 | 説明 |
|---|---|---|
| 1 | 出勤 | 通常出勤 |
| 2 | 出勤（休出） | 休日出勤 |
| 3 | 出勤（午前休） | 午前半休取得日 |
| 4 | 出勤（午後休） | 午後半休取得日 |

上記以外の勤務区分（公休、有休、欠勤 等）は突き合わせ対象外とする。

> ※ 対象区分は将来追加される可能性があるため、設定ファイル（環境変数またはJSON）で管理し、コードへのハードコードを避ける。

> ※ `segment_display_title` は日本語文字列で返却されることをAPIレスポンスで確認済み（2026-03-31）。

#### 3.2.6 ページネーション見積もり

| 項目 | 見積もり |
|---|---|
| 対象社員数 | 19名（2026-03時点） |
| 1ヶ月の勤怠レコード数 | 約372件（2026-03実績） |
| limit=100 でのページ数 | 4ページ |
| 月またぎの場合 | 最大8ページ（2ヶ月分） |
| ユーザー一覧 | 1ページ（19名 < limit=100） |
| 1回の実行あたり合計リクエスト数 | 約6〜10リクエスト（認証1 + 勤怠4〜8 + ユーザー1） |

#### 3.2.7 レート制限への対応

- 過去60秒のリクエスト累計で制限（具体的数値は非公開）
- 超過時: HTTP 429（レスポンスボディなし）
- 対策: ページネーションのリクエスト間に **1秒のウェイト** を挿入
- 429 受信時: **5秒待機してリトライ**（最大3回）

### 3.3 工数管理システム（SUMTIME）連携仕様

実機確認済み（2026-03-31）。

**データ取得方式（ハイブリッドアーキテクチャ）:**
- **社内PC**（`tools/fetch_sumtime_to_gcs.py`）がSSHトンネル経由でPostgreSQLに接続してデータ取得
- 取得したデータをJSON形式でCloud Storageにアップロード
- **Cloud Functions**はCloud StorageからJSONを読み込む（直接SSH接続しない）

**社内PC → SUMTIME DB 接続情報:**
- スキーマ: `sumtime`
- SSH認証: パスワード認証
- 接続情報: すべてSecret Managerで管理（セクション8.1参照）

#### 3.3.1 使用テーブル

```
working_achievements (9,529件) — 工数実績
  ├─ id (bigint, PK)
  ├─ working_user_id (bigint) → users.id
  ├─ working_schedule_id (bigint) → working_schedules.id
  ├─ start_at (timestamp) — 作業開始日時
  ├─ end_at (timestamp) — 作業終了日時
  └─ achievement_time (numeric) — 実績時間（時間単位の小数: 0.5=30分）

users (14件) — ユーザー
  ├─ id (bigint, PK)
  ├─ name (string) — 氏名
  ├─ email (string) — メールアドレス（= 突き合わせキー）
  └─ is_using (boolean) — 有効フラグ（trueのみ対象）

working_schedules (221件) — 作業項目（参考情報）
  ├─ id (bigint, PK)
  ├─ name (string)
  └─ project_id (bigint) → projects.id

projects (42件) — プロジェクト（参考情報）
  ├─ id (bigint, PK)
  ├─ name (string)
  └─ customer_id (bigint) → customers.id
```

#### 3.3.2 日別集計クエリ

```sql
SELECT
  u.email,
  u.name,
  DATE(wa.start_at) AS work_date,
  SUM(wa.achievement_time) * 60 AS total_minutes
FROM sumtime.working_achievements wa
JOIN sumtime.users u ON wa.working_user_id = u.id
WHERE u.is_using = true
  AND DATE(wa.start_at) BETWEEN :period_start AND :period_end
GROUP BY u.email, u.name, DATE(wa.start_at)
```

- `achievement_time` は時間単位の小数 → `* 60` で分に変換
- `is_using = false` のユーザーは除外
- 結果は email をキーとして日別の工数合計（分）を取得

### 3.4 社員マスタ（突き合わせ・通知用）

突き合わせキーの照合およびメール通知のために、メールアドレスと表示名の取得が必要。

| フィールド | 型 | 説明 |
|---|---|---|
| email | string | メールアドレス（= 突き合わせキー + 通知送信先） |
| display_name | string | 表示名（姓 + 名） |

## 4. 突き合わせロジック

### 4.1 対象期間

- **実行日の前日から `period_days` 日前まで**（デフォルト: 14日）
- 実行タイミング：日次モード（月〜金 9:00）または週次モード（毎週月曜 9:00）
- 例（日次・period_days=14）: 4/14(月)実行 → 3/31(月)〜4/13(日) が対象
- 例（週次・period_days=7）: 5/11(月)実行 → 5/4(月)〜5/10(日) が対象

### 4.2 集計単位

- 社員ごとに**日単位**で集計する
- 勤怠側：`segment_display_title` が突き合わせ対象の勤務区分（セクション3.2.5参照）の日の `actual_working_hours` を分に変換
- 工数側：`SUM(achievement_time) * 60` で日ごとの工数合計（分）を算出

### 4.3 差異の算出

```
差異（分） = 勤怠時間（分） − 工数時間（分）  ※日ごとに算出
```

### 4.4 通知閾値

- **|差異| ≧ 15分/日** の場合に通知対象とする
- 対象期間内に閾値超過日が1日でもあれば通知
- 差異の方向（勤怠 > 工数、勤怠 < 工数）も通知に含める

### 4.5 差異パターン

| パターン | 意味 |
|---|---|
| 勤怠 > 工数（≧ 15分） | 工数入力漏れの可能性 |
| 勤怠 < 工数（≧ 15分） | 打刻漏れ or 工数過大入力 |
| 勤怠あり・工数なし（SUMTIME登録あり） | 工数未入力 → 差異あり・通知 |
| 勤怠あり・工数なし（SUMTIME未登録） | 工数 = 0:00 として差異あり・通知 |
| 工数あり・勤怠なし | 勤怠未打刻 → 差異あり・通知 |
| 勤怠なし・工数なし（SUMTIME未登録） | 通知対象外（スキップ） |

### 4.6 通知動作（リマインダー方式）

- 差異が修正されるまで**繰り返し通知**する（日次モードは毎日、週次モードは毎週）
- 差異が解消された日は通知しない（差異なし = 通知なし）
- 週末（土日）は実行しない（日次モードの場合）

## 5. 通知仕様

### 5.1 通知チャネル

- 差異が検出された本人の**メールアドレス**に直接送信
- 管理者（村上勇治）に**サマリーメール**を送信（毎回実行後）
- **Microsoft Graph API（Mail.Send）** を使用（Azure AD アプリケーション権限）
- 送信元: `kintai-notice@use-eng.co.jp`（共有メールボックス・ライセンス不要）

**Graph API による送信フロー:**
```
1. OAuth 2.0 Client Credentials でアクセストークン取得
   POST https://login.microsoftonline.com/{tenant-id}/oauth2/v2.0/token

2. メール送信
   POST https://graph.microsoft.com/v1.0/users/{sender-email}/sendMail
   Body: {
     "message": {
       "subject": "【勤怠・工数差異】確認をお願いします",
       "body": { "contentType": "HTML", "content": "..." },
       "toRecipients": [{ "emailAddress": { "address": "{target-email}" } }]
     }
   }
```

**必要な Azure AD 設定:**

| 項目 | 内容 |
|---|---|
| アプリ名 | 勤怠工数通知システム（任意） |
| 種類 | アプリケーション権限（デーモン/サービス用） |
| 必要な権限 | `Mail.Send` |
| 認証フロー | OAuth 2.0 Client Credentials Flow |
| 管理者の同意 | 必要（グローバル管理者 or アプリケーション管理者） |

### 5.2 通知方式

- 差異が検出された社員ごとに **1件ずつ HTML メールを送信**
- **差異のある日のみ**を表示（差異がない日は表示しない）
- 差異が修正されるまで**繰り返しリマインド**（差異なし = 通知なし）
- 毎回実行後に**管理者へサマリーメールを送信**（通知人数・対象者一覧）
- 個人宛メールのため他のメンバーには内容が見えない（プライバシー保護）

## 6. 実行スケジュール

Cloud Scheduler に **3つのジョブ**を登録し、いずれか1つを有効化して運用する。

| ジョブ名 | cron式 | パラメータ | 初期状態 | 特徴 |
|---|---|---|---|---|
| `daily-check` | `0 9 * * 1-5` | `period_days: 14` | **有効** | 月〜金 毎日 9:00・差異を毎日リマインド |
| `weekly-check` | `0 9 * * 1` | `period_days: 7` | 無効 | 毎週月曜 9:00・週次で前週分をチェック |
| `monthly-check` | `0 9 1 * *` | `period_month: 前月` | 無効 | 毎月1日 9:00・前月1日〜末日をチェック |

> ※ `monthly-check` の `period_month` は実行時に前月の年月（例: 3月1日実行 → `"2026-02"`）を動的に生成する。

**切り替え手順（GCPコンソール → Cloud Scheduler、または管理UIから操作）:**
1. 現在有効なジョブを選択 → 「無効化」
2. 使いたいジョブを選択 → 「有効化」

| 項目 | 日次モード | 週次モード | 月次モード |
|---|---|---|---|
| 実行日 | 月〜金（祝日除く） | 毎週月曜（祝日除く） | 毎月1日 |
| 対象期間 | 実行日前日〜14日前 | 実行日前日〜7日前 | 前月1日〜末日 |
| リマインド頻度 | 毎日 | 週1回 | 月1回 |

| 項目 | 値 |
|---|---|
| 実行時刻 | 9:00 JST |
| タイムゾーン | Asia/Tokyo |
| 祝日判定 | 実行時に日本の祝日かどうかを判定し、祝日ならスキップ（即終了） |
| 祝日判定方法 | Python `jpholiday` ライブラリを使用 |

**即時実行（臨時チェック）:**

管理UIの「即時実行」画面から Cloud Functions を呼び出す。curl コマンドによる直接呼び出しも可能（セクション5.5参照）。

## 7. エラーハンドリング

### 7.1 エラー通知先

システムエラー発生時は、管理者（村上勇治 `y-murakami@use-eng.co.jp`）に**メール通知** + Cloud Loggingにエラーログを記録する。

エラー通知メールイメージ:
```
件名: 【エラー】勤怠・工数突き合わせシステム 異常検知

以下のエラーが発生しました。

実行日時: 2026/04/14 09:00
エラー種別: HRMOS API接続失敗
詳細: Connection timeout after 3 retries
```

### 7.2 API接続エラー

| エラー種別 | 対応 |
|---|---|
| HRMOS API 接続失敗 | 最大3回リトライ（指数バックオフ）。失敗時は管理者にメール通知 |
| SUMTIME DB 接続失敗 | 最大3回リトライ（指数バックオフ）。失敗時は管理者にメール通知 |
| Graph API（メール送信）失敗 | 最大3回リトライ（指数バックオフ）。HTTP 429 時は Retry-After に従う。失敗時はCloud Loggingにエラーログ記録 |
| レート制限（HTTP 429） | 5秒待機してリトライ（最大3回） |

### 7.3 部分障害時の動作

- HRMOS API または Cloud Storage（SUMTIMEデータ）のどちらか一方でも取得に失敗した場合、**処理全体を中止**する
- 片方のデータだけでは正確な突き合わせができないため、不完全な通知は行わない
- 中止時は管理者にエラーメールを送信し、翌営業日の実行で再取得する
- メール送信中に一部送信が失敗した場合は、失敗分のみCloud Loggingに記録し、次回実行で再通知される（リマインダー方式のため自動リカバリ）

### 7.4 データ異常

| 異常種別 | 対応 |
|---|---|
| HRMOS打刻あり・SUMTIME未登録 | 工数 = 0として差異検出・通知（セクション4.5参照） |
| HRMOS打刻なし・SUMTIME未登録 | 通知対象外としてスキップ |
| 勤怠時間が24hを超過 | 異常値として警告付きで管理者通知 |
| 工数時間が負の値 | 異常値として警告付きで管理者通知 |

### 7.5 ログ出力

- すべての実行結果を Cloud Logging に出力
- ログレベル：INFO（正常）、WARNING（データ異常）、ERROR（API障害）

## 8. セキュリティ

### 8.1 認証情報の管理（Secret Manager）

全認証情報を **3つのJSONシークレット** でSecret Managerに管理する。
GCPの無料枠（アクティブバージョン数6以下）に収まるよう、システム単位でグルーピングしている。

| シークレット名 | 内容（JSON形式） |
|---|---|
| `hrmos-credentials` | `{"secret_key": "..."}` — HRMOS API Secret Key（Base64済み） |
| `sumtime-credentials` | `{"ssh_host":"...","ssh_user":"...","ssh_password":"...","db_host":"...","db_name":"...","db_user":"...","db_password":"..."}` |
| `azure-credentials` | `{"tenant_id":"...","client_id":"...","client_secret":"..."}` — Azure AD 認証情報 |

> ※ Secret Managerの無料枠は「アクティブなシークレットバージョン数6以下」。3シークレット × 1バージョン = 3で無料枠内。
> ※ コード内では `load_all_secrets()` が3シークレットを取得してJSON展開し、旧来の11キー平坦dictに変換して返すため、呼び出し元のコード変更は不要。
> ※ Azure AD アプリに必要な権限は `Mail.Send`（アプリケーション権限）のみ。

### 8.2 通信

- すべての外部API通信は HTTPS
- Cloud Functions には最小権限の IAM ロールを付与

### 8.3 データ保護

- 個人情報（氏名、メールアドレス）を扱うため、Cloud Logging・管理UIのアクセス権限は管理者に限定
- BigQuery導入時（将来機能）はアクセス権限を管理者に限定すること
- ログに認証情報を出力しない

## 付録B: Teams通知を断念した経緯（2026-04-20）

### 背景

当初仕様ではMicrosoft Graph APIを使用したTeams個人チャット（DM）送信を採用していた。  
2026-04-20に実機検証を行ったところ、以下の制約により断念した。

### 検証結果

**試みた方式:** Microsoft Graph API（Application permissions / Client Credentials Flow）

| ステップ | 結果 | 備考 |
|---|---|---|
| アクセストークン取得 | ✓ 成功 | |
| ユーザーID取得 | ✓ 成功 | |
| サービスプリンシパルID取得 | ✓ 成功 | |
| 1:1チャット作成 | ✗ HTTP 403 | AclCheckFailed |
| グループチャット作成 | ✗ HTTP 403 | OperationFailed |

### 失敗の原因

Microsoft Graph APIのアプリケーション権限（サービスプリンシパル）には以下の制約がある:

1. **サービスプリンシパルは1:1チャット（oneOnOne）のメンバーになれない**  
   → `AclCheckFailed` エラー

2. **グループチャット作成にはユーザーメンバーが最低2名必要**（サービスプリンシパルはカウント外）  
   → 1名の社員への通知には対応不可

### 代替案の検討

| 案 | 結論 |
|---|---|
| グループチャット（ユーザー2名）| 管理者がすべての会話を見ることになりプライバシー上不適切 |
| **Azure Bot Service** | Azure サブスクリプションが必要。現環境では取得していないため断念 |
| 専用通知アカウント（M365ライセンス） | 月額約540円のライセンスコストが発生するため断念 |
| **メール通知（Graph API Mail.Send）** | **採用**。Application permissionsで確実に動作。追加コストなし |

### 将来のTeams通知移行について

Azureサブスクリプションを取得した場合、以下の方式でTeams DM通知への移行が可能。

1. Azure Bot Service（F0無料プラン）でボットを登録
2. Teamsアプリマニフェストを作成し組織カタログに登録
3. Graph APIで全社員にボットを一括インストール（`TeamsAppInstallation.ReadWriteForUser.All`）
4. インストール済みチャット経由でメッセージ送信

この方式は今後のTeams通知集約基盤としても活用できる。
